产品向

众筹事件的收获以及对产品设计的深思

事件回顾

没想到仅仅半天时间，就完成了人生第一笔众筹。在这短短的12小时，我收获了旁人对我的鼓励、支持、不解、质疑，但这一切，对我来说都是一笔难得的宝贵财富。可能大家最疑惑的是，为啥我连288软妹币都需要众筹呢？下面就为大家揭开谜底：

其实作出这一决定，是因为我偶然翻看到一年前微信的一个活动：

微信官方发布了一本新书－－《微信思维》，但购买方式很特别：通过转发购买页面到朋友圈或者好友，然后看好友是否愿意用自己的微信红包资助一部分书价，当筹得的总价达到书价后，即可认定为购买成功。

这不就是赤裸裸的众筹吗？ヾ( ´ー｀)ﾉ

这激起了我极大的好奇心，再加上最近看的好多文章都在讨论众筹的各种各样，也许，我也能自己尝试下发起一次众筹？然后我竟然立马开始计划起来了（这里竟然一词用得不能再准确，我这人属于拖延症末期人群，脑里曾有无数想法，都被我内心的拖延症小人扼杀在摇篮里了，而这次，我竟然会立马行动，我事后回想，老天都被我自己感动得哭了(∩∀｀●)っ）

众筹圈内有句颇值得玩味的话－－国外看情怀，国内看人脉。对于这种检验人脉的危险游戏，我应该找谁来放下尊严陪我疯一把呢？当我茫然地滚动着手机通讯录20多个来回后，蓦然发觉，对于此等屁事，最适合被坑的人选，还是－－自己。

解决了人选问题后，又有新一轮问题浮出。那就是，众筹的额度该如何把握？额度高了容易吓跑参与者，额度低了又会被人误以为是在化缘。冥思苦想期间，我又收到了来自我超级喜欢可一直不舍得买的写作软件－－Ulysses的博客订阅邮件，看着那¥283的定价，我意识到，也许，288是个完美的额度（我纯粹是考虑到，中国人，尤其广东人，最讲究数字的吉利了，真心并非故意想多赚5块钱的(੭ु•́ω•̀)੭ु⁾⁾）

于是，众筹事件就这样，在偶然与必然的交织下，拉开了序幕。

收获

OK，鉴于最终结果大家都知道了，我不想过于矫情地去描述众筹过程有多么的惊心动魄与波澜壮阔，今天我更想谈的是，这一事件给我带来的收获。

收获1 关于如何激励人们自愿按照你的意愿去采取行动的方法

上面那句话有点长，但真心不啰嗦（我真的用心精简了好久的好吗ヾ(╹ᆺ╹๑)），我这次众筹只发送到了5个群，其中有2个群里的细心的小伙伴可能会发现，众筹文案其实是有2个版本的（具体如下）：
第一个版本大意：

我想要一个叫Ulysses的软件（what）》需要大家帮忙众筹（how）》圆我写作梦（why）

第二个版本大意：

圆我写作梦（why）》需要大家帮忙众筹（how）》我想要一个叫Ulysses的软件（what）

可以看到，除了陈述顺序，具体内容是基本一样的（当然，第二个版本由于是再次编写，所以对比第一个版本，文字上会有适当的润色），但第一个版本发布时，我收到了一大波伴随着口水飞溅的质疑声，大致如下：

  1. 这是什么鬼软件？ （确实很难三言两语说清楚）
  2. 这是什么鬼新型骗术？（逼迫我自报宿舍门牌号与室友）
  3. 这是什么鬼同窗？（潜台词：怎么3张毛爷爷都要众筹，看来混得不太好）

归纳起来，就是对物－对事－对人。下面来说说两个版本给我带来的效果。第一个版本，为我筹得12元；第二个版本，为我带来309元。是什么原因导致这个巨大差异呢？

其实我只是验证了一下从TED上学到的一个理论－－黄金圆环（出自《how great leaders inspire action》——by simon sinek），即人们在解释某一件事情时，往往习惯按照what－how－why这样的顺序去陈述，但其实，更有效的陈述顺序是why－how－what，这个演讲对我影响深刻，因为解答了我心底长久以来的一个疑问，就是乔布斯哥们在Apple发布会上介绍新产品（从iPod到iPhone，iPad无一例外）出场前总要花上好几分钟强调一下，市场上还有哪些产品空间，人们的需求还有哪些待挖掘的部分，模拟一下用户的使用场景，陈述完这一切以后，才把旗舰产品show给各位看官过目，继而介绍产品特点，功能，配置啥的。

这做法的好处其实十分明显，充分利用了一个心理学效应－－首因效应，如果一开始抛出产品（what）的话，那么用户要衡量的就是，是否愿意接受这个产品（这可是涉及到真金白银啊，必须慎重），但如果一开始，从why入手，那么，用户要衡量的就是，我是否愿意接受这个理念（理念可是免费的呀亲），一旦切合用户的三观，激起用户的共鸣，那么这个理念就算是植入成功了（比起小李子在《盗梦空间》里的折腾，这可省事儿多了好嘛）。

具体到我这个实例，第一个版本，我企图向用户推销Ulysses这款软件（这应该是看到第一个版本的看官最大的感受），从而希望通过众筹获得这款优秀的软件，于是，在用户眼里，我更像是一个在百货商场的玩具区不顾爹娘颜面、死活赖着不走、在地上奋力打滚只为让爹娘掏钱买一个不知道有啥用处的破玩意的熊孩子；而第二个版本，一开始，我就抛出一个理念，请帮助我，圆我写作梦，哇塞，在用户眼中，我突然就升级为一个命途坎坷郁郁不得志、而心中梦想却从未熄灭的八分青年好嘛！（ps：不过现实中我真的很喜欢写东西的啦）

总结，推销一个理念比直接推销产品来的容易，于是，产品设计过程中，除了保证产品的质量以外，为产品包装一个切合目标客户三观与需求的理念更显重要。

收获2 众筹新玩法：亮点在众，不在筹

对于众筹这一新兴词汇，很多人都会不自觉地“望文生义”，把理解重点落在“筹”上，因此又被戏称为“屌丝式集资”，而我经过这一次“实地考察”，发现，有不少看到我的众筹信息的朋友，都百度了一下Ulysses到底是何方神圣，甚至直接私信我，这软件有啥过人之处，为啥值得我拉下自尊为之筹集3百大洋，这结果是稍微出乎我意料的，不过正是这个有趣的现象，让我意识到，众筹，原来还可以这么玩：

将众筹作为产品宣传or营销的方式。

对于朋友圈、微博等渠道来说，众筹与其他硬广告相比，有一个天然优势：

用户具有天然的传播动机－－因为用户在传播中能获利

正如本文开头所提及的微信团队举办的《微信思维》众筹活动，也许用户对微信思维并非有多么饥渴的研究欲望，但看到只需要手指点一点转发，就可以让好基友帮忙付款，顺利的话，就能免费获得一本价值68 RMB的高逼格书籍来装装X，即使不顺利，自己也没吃亏啊，然后这众筹信息就像爆炸性新闻一样，一传十，十传百，尤其是“懂行识货”者的协助，如果他们在自己朋友圈里转发，看到的人都会因为“识货者”的权威性，也将纷纷跟风，事态如斯发展，最终大赢家，并不是众筹成功的参与者，而是名利双收的众筹活动策划者。

小彩蛋： 我觉得我这次为Ulysses团队作了一次免费的推广，于是半开玩笑地写了一封邮件给开发商soulmen邀功，谁知客户经理竟然真回信了，他说，很赞赏我为他们软件推广作出努力，不过还是不能在我购买时给我打折哟（歪果仁，真抠门～～）

总结，互联网时代，要有互联网思维，众筹着眼于众，借参与者之手，铺开知名度，利的筹得，自然是水到渠成的事。

收获3 用户都是值得我们细心呵护的懒癌晚期患者

我觉得这次我使用的众筹服务（由某微信公众号所提供，具体就不说了，不然有抹黑嫌疑）有一系列让人绝望的缺陷，极大地暴露了其用户意识的缺乏（又或者只是技术门槛无法跨越？who knows～），同时这些缺陷也是参与者投诉最多的点，具体摘录如下：

  1. 只能用支付宝，不能用微信红包（致命的）；
  2. 必须手机账号注册（这点将在下一节展开讨论）；
  3. 手机无法收到验证码（看来众筹服务商跟三大运营商关系没打好啊）；
  4. 网页无法加载（网页设计优化不到位）；
  5. 找不到付款按钮（这－。－也许不能全怪网页设计者吧）；

以上这些点都极大地打击了用户的参与积极性（写到这里，我真的不得不给各位参与了众筹帮助我顺利买到我正用着它来感受码字快感的Ulysses的亲们磕个头了，那么鬼麻烦的参与门槛你们都能一一克服，只为满足我那小小的愿望，不说了，忍不住了，我去擦擦鼻涕(ಡ౪ಡ)）

那到底以上的繁琐门槛对众筹效果产生多大影响呢？在我收到第n＋1位朋友向我提出相关控诉之后，我又在每个群上发了一条信息：

给我发微信红包也是可以的哟～

结果，让我惊讶了，短短半小时，我就获得了价值¥155的私人微信红包，其中不乏久未露面出国多时的异地友人，这才让我惊叹，在微信平台搞众筹竟然不支持微信红包是多么愚蠢的设定！ 然后我粗略计算了一下单位时间收入：
备注：即用不同途径所获得的收入除以所花费的总时间）
结果显示，每减少一步操作，单位时间收入可增长18倍！（样本量不足，仅供参考）

反过来讲，操作的繁琐步骤，可以让用户参与的积极性成指数式递减。

总结，若想用户顺利参与到我们组织的活动里，参与门槛以及中间环节都应尽量精简，用户没那么多耐心跟你耗，而诸如“一键（步）即达”的操作方式值得我们去探究，如果能做到，用户不需动用太多肌肉群，花费太多ATP，只是安详地躺在床上也能尽情享受到我们的服务，那就是极好的。

收获4 关于充盈新用户蓄水池的做法优劣的思考

其实事后很多朋友跟我吐槽，说响应朋友号召众筹一下都需要用手机号码注册账户，真无耻。我后来想想，这无非是众筹服务商为了招揽新用户所采取的比较拙劣的手段罢了，即使用户参与了，注册了，也难免因为对服务商的前置的厌恶感而“怒取关”，这“口碑”传开了，也就得不偿失了。

其实，要衡量扩大用户蓄水池的各种推广手段是否有效，可以采用以下两个标准去衡量：

  1. 客户生命周期价值（CLV，即与客户保持联系期间从每个客户赚取的总盈利）
  2. 客户获取成本（CAC，赢得新客户的平均成本）

若CLV/CAC的比值越大，证明推广方式越有效。按此公式推算，上述众筹网虽然在CAC上是免费的，不过CLV也注定不会高到哪里去，所以，用户蓄水池很快就枯干了，最后只剩发烧友了（ps：这让我想起了这几年来一直微软抛弃却一直对微软不离不弃的windows phone用户啊，真是可歌可泣的爱情故事，如果你身边有人还在用WP系统，那人肯定是个长情货，请毫不犹豫嫁给他／娶了她吧！）

而较让我印象深刻的，有两家公司的推广手段：

  1. Paypal（国外类似于支付宝的存在），可归纳为“舍不得孩子套不住狼”法，具体做法是，新客户首次注册，就会获得10美元，然后大家又纷纷向好基友推荐使用Paypal（毕竟这赚钱效率特高啊！），很快，用户蓄水池就成指数式增长了。
  2. Kipling，专营女士包包的，我将它的推广营销手法归纳为，“好闺蜜”法，具体手段是，女孩们每一次官网下单之后，都弹出一个框框，请求填写自己最好的闺蜜的电邮地址（前提是，对方还没是会员），如果这笔订单交易成功，闺蜜就会收到价值15美元的代金券。

上述公司的做法，虽然前期CAC投入比较凶残，但正因为有良好的可持续性（上述两种方式都具有病毒性营销特质），所以能保证CLV的稳步增长，最后还是会以盈利收场。

总结，推广模式选择时不能只短视地顾及投入成本，应从CLV/CAC的比值出发，选取最适合的推广途径。

极客向

用hexo在github上构建博客——Mac版

第0篇博客

历尽千辛万苦终于搭建好了这么一个简陋的博客，其实原本应该半小时内就可以实现的工作，结果耗了我4个小时（都够我看两部大片了），主要有两方面原因，其中之一是，我真的只是个小白；而第二呢，也是最最主要的原因，就是网上的教程良莠不齐（ 基本上莠者居多，且几乎都是“引用”自某极客2年前的一篇教程，说白了就是相互抄袭，甚至连原作者的用户名都没改，这写作态度还好意思开博客也是让人微醺），难得找到一两篇良的，结果都是基于windows系统的，连Linux都有（为何要歧视水果用户٩(ˊ࿀ˋ⋆)و﻿），于是我当初就下定决心，待我搭建好了博客，得将我这苦逼心路历程（不敢说是教程，免得被极客们口水淹了）都写下来，希望能帮助有同样建博客需求的Mac用户少走一些弯路。事不宜迟，即将开始：

以下步骤主要参考 hexo.io 以及 help.github.com

安装xcode

这个比较方便，直接在Mac App Store上搜索下载就ok，下载好了以后，启动并进入 Preferences -> Download -> Command Line Tools -> Install 安装命令行工具。

安装Git

使用Homebrew来安装即可。启动终端机(即terminal)，将下列代码：
ruby -e "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/master/install)"
黏贴进去，回车，稍等片刻，完成。

安装Node.js

官网推荐的安装方式是使用nvm。以下安装脚本二选一即可：
cURL:
$ curl https://raw.github.com/creationix/nvm/master/install.sh | sh

Wget:
$ wget -qO- https://raw.github.com/creationix/nvm/master/install.sh | sh

安装完成后，重启终端机并执行下列命令即可：
$ nvm install

关于nvm其他安装方式的详情可点击：https://github.com/creationix/nvm

安装Hexo

所有准备工作已搞定，终于来到高潮啦！正式可以使用npm来安装Hexo！只需输入：
$ npm install -g hexo-cli

安装成功后再输入下面的：
$ npm install -g hexo

剩下的只是等待，让它自动安装完毕即可。

建站

安装Hexo后，又要再次打开终端机啦，我们需要让Hexo在指定文件夹中新建所需要的文件哟，建议是直接在finder中的个人专属(就是图标是那间小屋子，名字是你为你心爱的Mac改的名字哟)新建一个名为“hexo”的文件夹即可，然后输入下列代码：
$ cd /hexo (这里的hexo是表示新建的文件夹名字) ``$ hexo init
$ npm install

新建完成以后，指定文件夹的目录如下：

├── _config.yml
├── package.json
├── scaffolds
├── scripts
├── source
| ├── _drafts
| └── _posts
└── themes

其中网站具体的配置，是打开config.yml 直接修改的。(这文件可用xcode作为默认打开应用程序直接打开的哟，并不用像大部分博客说的，下载啥sublime text的，毕竟st正版太贵，至于破解版……个人感觉，咱们是时候学会尊重应用制作者的知识产权了)

具体的配置方法，请前往http://hexo.io/zh-cn/docs/configuration.html慢慢参详，都是比较简单的啦。

部署github

下面就到了如何将hexo构建的静态博客部署到github上啦，现在已经完成了50%啦，keep moving！！下面是具体做法：

  1. 注册github；
  2. 新建仓库repository，且仓库名必须为yourUserName.github.com 或者yourUserName.github.io；
  3. 最最难最最麻烦的一步，生成并添加SSH key。下面详细来说说：

Step 1: Check for SSH keys

首先呢，不要太冲动急着生成SSH keys，首先得查一查自己的Mac里有没有现成的嘛。如何查询呢，启动终端机，输入以下代码并回车：
$ ls -al /.ssh

Step 2: Generate a new SSH key

如果木有的话呢，就要亲自来生成啦！输入以下代码，用自己的github注册邮箱来生成：
ssh-keygen -t rsa -C "your email@example.com"

然后会出现以下代码：
Enter file in which to save the key (/Users/you/.ssh/id rsa):
官方建议是直接按enter即可；

接着你要输入一个密码：
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
ps: 密码别起得太简单哟！

输入结束以后，你会获得一个指纹啦，id啦，还有你的SSH key啦，具体长啥样呢？可以参考下图：
Your identification has been saved in /Users/you/.ssh/id rsa.
Your public key has been saved in /Users/you/.ssh/id rsa.pub.
The key fingerprint is:
01:0f:f4:3b:ca:85:d6:17:a1:7d:f0:68:9d:f0:a2:db your email@example.com

Step 3: Add your key to the ssh-agent

现在需要把SSH key 添加到ssh-agent啦，首先需要确认一下ssh-agent是否已启用：
$ eval "$(ssh-agent -s)"
Agent pid 59566

把生成的SSH key添加至ssh-agent：
$ ssh-add /.ssh/id rsa

Step 4: Add your SSH key to your account

登录你的github账户，前往Account settings -> SSH Keys -> Add SSH Key， 把刚生成的文件：idrsa.pub里面的内容一个字符不漏地复制到key里面，然后点击 Add key，最后输入你的github的密码即可。

Step 5: Test the connection

最后来测试一下是否成功连接啦，打开终端机，输入：
$ ssh -T git@github.com

然后会看到一大串警告，忽略，直接输入： yes ，回车；

如果你看到以下信息：
Hi username! You've successfully authenticated, but GitHub does not provide shell access.

且里面的username是你的账户名字，那证明链接大成功！！

生成静态页面

启动终端机，输入：
$ cd /hexo (这里的hexo是表示安装hexo的文件夹名字) ``$ hexo generate

本地启动

执行以下命令，启动本地服务，即可预览文章，进行调试啦
$ hexo server

浏览器键入 http://localhost:4000 可观察到效果。

将hexo构建的博客与github建立关联

  1. 安装 hexo-deployer-git 。终端机输入：
    $ npm install hexo-deployer-git --save

  2. 修改config.yml设置：
    用xcode打开文件，按以下项目修改：
    deploy:
    type: git
    repo: <repository url> (从github账户可查询到，建议使用https) `` branch: (一般来说，是master，也可从github账户查询到)

  3. 保存退出；

  4. 在终端机输入以下代码：
    $ hexo deploy
    如果显示done的话，那就圆满啦！！我看到成功信息的时候，真有种想哭的冲动啊！
